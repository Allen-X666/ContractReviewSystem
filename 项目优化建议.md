# 合同审查系统优化建议

## 项目概述

本项目是一个智能合同审查系统，采用前后端分离架构：
- **前端**：Vue 3 + Element Plus + Vite
- **后端**：Spring Boot 4.0.5 + MyBatis + MySQL + Redis
- **AI服务**：通过 FastAPI 调用 Python AI 服务进行合同审查

---

## 一、后端优化建议

### 1.1 线程池管理优化

**问题**：
- `ReviewServiceImpl` 中使用了 `Executors.newScheduledThreadPool(10)` 创建线程池，但没有正确关闭
- 异步保存审查记录时，异常处理不够完善

**建议**：
```java
// 使用 Spring 管理的线程池，而不是手动创建
@Configuration
public class ThreadPoolConfig {
    @Bean("reviewTaskExecutor")
    public ThreadPoolTaskExecutor reviewTaskExecutor() {
        ThreadPoolTaskExecutor executor = new ThreadPoolTaskExecutor();
        executor.setCorePoolSize(10);
        executor.setMaxPoolSize(20);
        executor.setQueueCapacity(100);
        executor.setThreadNamePrefix("review-task-");
        executor.setRejectedExecutionHandler(new ThreadPoolExecutor.CallerRunsPolicy());
        executor.setWaitForTasksToCompleteOnShutdown(true);
        executor.setAwaitTerminationSeconds(60);
        executor.initialize();
        return executor;
    }
}
```

### 1.2 SSE 连接优化

**问题**：
- `connectProgress` 方法中每个连接都创建新的 `ScheduledExecutorService`，可能导致资源泄漏
- 缺少连接数限制，可能被恶意攻击

**建议**：
```java
// 1. 使用统一的线程池管理 SSE 连接
// 2. 添加连接数限制
@Component
public class SseConnectionManager {
    private final ConcurrentHashMap<Integer, SseEmitter> emitters = new ConcurrentHashMap<>();
    private final AtomicInteger connectionCount = new AtomicInteger(0);
    private static final int MAX_CONNECTIONS = 100;
    
    public SseEmitter createEmitter(Integer reviewId) {
        if (connectionCount.get() >= MAX_CONNECTIONS) {
            throw new BusinessException("连接数超限，请稍后重试");
        }
        // ... 创建逻辑
    }
}
```

### 1.3 缓存策略优化

**问题**：
- Redis 缓存没有设置过期时间，可能导致内存无限增长
- 缓存 key 的命名不够规范

**建议**：
```java
// 统一缓存管理
@Component
public class ReviewCacheManager {
    private static final long REVIEW_RESULT_TTL = 24; // 24小时
    
    public void cacheReviewResult(Integer reviewId, ReviewResultVO result) {
        String key = CacheKeys.REVIEW_RESULT + reviewId;
        redisTemplate.opsForValue().set(key, result, REVIEW_RESULT_TTL, TimeUnit.HOURS);
    }
}
```

### 1.4 数据库查询优化

**问题**：
- `getContractList` 方法中，先查询合同列表，再循环查询审查ID，存在 N+1 查询问题
- 缺少必要的索引

**建议**：
```sql
-- 添加复合索引优化查询
CREATE INDEX idx_contract_user_id_status ON contract(user_id, review_status);
CREATE INDEX idx_review_task_contract_user ON review_task(contract_id, user_id, created_at);
```

### 1.5 事务边界优化

**问题**：
- `startReview` 方法事务范围过大，包含外部 API 调用
- 异步方法中的事务处理可能存在问题

**建议**：
```java
@Override
public Result<Map<String, Object>> startReview(String authorization, StartReviewDTO dto) {
    // 1. 先调用外部 API（不在事务中）
    FastApiResult<Map<String, Object>> fastApiResult = fastApiClient.startReview(dto);
    
    // 2. 异步保存记录（独立事务）
    if (fastApiResult.isSuccess()) {
        asyncSaveReviewRecord(userId, dto.getContractId(), fastApiResult.getData());
    }
}

@Transactional(propagation = Propagation.REQUIRES_NEW)
public void asyncSaveReviewRecord(...) {
    // 独立事务保存记录
}
```

### 1.6 配置安全优化

**问题**：
- `application.yml` 中敏感信息（密码、密钥）硬编码
- 邮箱密码明文存储

**建议**：

### 1.7 API 设计优化

**问题**：
- 部分接口返回数据量过大，没有分页
- 缺少接口限流

**建议**：
```java

@RateLimiter(key = "review:start", permitsPerSecond = 10)

public Result<Map<String, Object>> startReview(...) {}

// 2. 大数据量接口添加分页
@GetMapping("/history")
```

## 二、前端优化建议

### 2.1 状态管理优化

**问题**：
- `review.js` store 中，SSE 连接逻辑和状态管理耦合
- 缺少连接状态管理，页面刷新后丢失审查状态

**建议**：
```javascript
// 使用 localStorage 持久化审查状态
export const useReviewStore = defineStore('review', () => {
  // ...
  
  // 持久化当前审查状态
  function persistReviewState() {
    const state = {
      reviewId: currentReview.value?.reviewId,
      status: reviewStatus.value,
      progress: reviewProgress.value
    }
    localStorage.setItem('review_state', JSON.stringify(state))
  }
  
  // 恢复审查状态
  function restoreReviewState() {
    const saved = localStorage.getItem('review_state')
    if (saved) {
      const state = JSON.parse(saved)
      // 重新连接 SSE
      if (state.status === 'processing') {
        connectProgress(state.reviewId)
      }
    }
  }
})
```

### 2.3 错误处理优化

**问题**：
- API 错误处理分散在各个组件中
- 缺少统一的错误提示和重试机制

**建议**：
```javascript
// request.js 中添加统一错误处理
service.interceptors.response.use(
  response => response,
  error => {
    const { config, response } = error
    
    // 自动重试机制
    if (!config.__retryCount && config.method === 'get') {
      config.__retryCount = 1
      return service(config)
    }
    
    // 统一错误提示
    const message = response?.data?.message || '网络错误，请稍后重试'
    ElMessage.error(message)
    
    return Promise.reject(error)
  }
)
```

### 2.4 文件预览优化

**问题**：
- PDF 预览使用 iframe，加载大文件时体验不佳
- 没有文件缓存机制

**建议**：
```javascript
// 使用 PDF.js 分片加载
import * as pdfjsLib from 'pdfjs-dist'

// 实现分片加载和缓存
class PDFPreviewManager {
  private cache = new Map()
  
  async loadDocument(url) {
    if (this.cache.has(url)) {
      return this.cache.get(url)
    }
    
    const loadingTask = pdfjsLib.getDocument({
      url,
      rangeChunkSize: 65536, // 64KB 分片
    })
    
    const pdf = await loadingTask.promise
    this.cache.set(url, pdf)
    return pdf
  }
}
```

## 三、架构优化建议

### 3.1 引入消息队列

**问题**：
- 合同审查是耗时操作，同步调用可能导致超时
- 审查任务没有持久化，服务重启会丢失

**建议**：
```java
// 使用 RabbitMQ 或 Kafka 处理审查任务
@Service
public class ReviewTaskService {
    @RabbitListener(queues = "review.queue")
    public void handleReviewTask(ReviewTaskMessage message) {
        // 异步处理审查任务
        // 1. 调用 FastAPI 发起审查
        // 2. 轮询进度
        // 3. 完成后通知用户
    }
}
```

### 3.2 分布式锁

**问题**：
- 同一合同可能被同时发起多次审查
- 并发情况下可能出现数据不一致

**建议**：
```java
@Service
public class ReviewLockService {
    public boolean tryLock(Integer contractId) {
        String key = "lock:review:" + contractId;
        Boolean locked = redisTemplate.opsForValue()
            .setIfAbsent(key, "1", 5, TimeUnit.MINUTES);
        return Boolean.TRUE.equals(locked);
    }
}
```

### 3.3 服务熔断降级

**问题**：
- FastAPI 服务故障时，没有降级策略
- 大量请求可能压垮后端服务

**建议**：
```java
// 使用 Resilience4j 实现熔断
@CircuitBreaker(name = "fastapi", fallbackMethod = "fallbackStartReview")
public FastApiResult<Map<String, Object>> startReview(StartReviewDTO dto) {
    return fastApiClient.startReview(dto);
}

public FastApiResult<Map<String, Object>> fallbackStartReview(StartReviewDTO dto, Exception ex) {
    log.warn("FastAPI 服务不可用，使用降级策略");
    // 返回队列中等待处理
    return FastApiResult.success(Map.of("status", "queued"));
}
```

### 3.4 数据一致性保障

**问题**：
- 审查状态在多个地方维护（Redis、数据库、FastAPI）
- 可能出现状态不一致

**建议**：
```java
// 使用事件驱动架构保证最终一致性
@EventListener
public void onReviewCompleted(ReviewCompletedEvent event) {
    // 1. 更新数据库状态
    reviewMapper.updateStatus(event.getReviewId(), "completed");
    
    // 2. 更新缓存
    redisTemplate.delete(CacheKeys.REVIEW_RESULT + event.getReviewId());
    
    // 3. 发送通知
    notificationService.sendReviewCompleteNotification(event.getUserId());
}
```

## 四、安全优化建议

### 4.1 文件上传安全

**问题**：
- 文件类型校验仅通过后缀名判断，不安全
- 没有文件大小和内容的深度检查

**建议**：
```java
@Service
public class FileSecurityService {
    public void validateFile(MultipartFile file) {
        // 1. 校验 MIME 类型
        String contentType = file.getContentType();
        if (!ALLOWED_TYPES.contains(contentType)) {
            throw new BusinessException("不支持的文件类型");
        }
        
        // 2. 校验文件头（魔数）
        byte[] header = file.getBytes(0, 8);
        if (!isValidFileHeader(header)) {
            throw new BusinessException("文件格式异常");
        }
        
        // 3. 病毒扫描（集成 ClamAV）
        virusScanService.scan(file);
    }
}
```

### 4.2 SQL 注入防护

**问题**：
- 部分动态 SQL 使用字符串拼接
- MyBatis 参数化查询需要全面检查

**建议**：
```xml
<!-- 使用 #{} 占位符，避免 ${} -->
<select id="getContractHistoryList" resultType="ReviewHistoryVO">
    SELECT * FROM review_task
    WHERE user_id = #{userId}
    <if test="keyword != null and keyword != ''">
        AND contract_name LIKE CONCAT('%', #{keyword}, '%')
    </if>
</select>
```

### 4.3 敏感数据脱敏

**问题**：
- 日志中可能打印敏感信息
- 接口返回包含用户隐私数据

**建议**：
```java
// 日志脱敏
@Slf4j
public class SensitiveDataFilter {
    public static String maskPhone(String phone) {
        return phone.replaceAll("(\\d{3})\\d{4}(\\d{4})", "$1****$2");
    }
}

// 接口响应脱敏
@JsonSerialize(using = MaskPhoneSerializer.class)
private String phone;
```

## 六、优先级排序

### 高优先级（立即执行）
1. **配置安全**：移除硬编码密码，使用环境变量
2. **线程池管理**：修复线程池泄漏问题
3. **SSE 连接限制**：防止连接数过多导致服务崩溃
4. **数据库索引**：优化慢查询

### 中优先级（近期执行）
1. **缓存策略**：添加过期时间和缓存管理
2. **前端状态持久化**：防止页面刷新丢失审查状态
3. **文件上传安全**：添加文件类型和内容校验
4. **代码分割**：优化前端构建产物

### 低优先级（长期规划）
1. **消息队列**：引入 RabbitMQ 处理异步任务
2. **服务熔断**：集成 Resilience4j
3. **分布式锁**：防止重复审查
4. **应用监控**：集成 Prometheus + Grafana

---

## 七、总结

本项目整体架构清晰，代码规范良好，但在以下方面需要重点关注：

1. **资源管理**：线程池、SSE 连接需要更好的生命周期管理
2. **安全防护**：敏感配置、文件上传需要加强安全校验
3. **性能优化**：数据库查询、前端渲染需要针对性优化
4. **高可用**：引入消息队列、熔断降级提升系统稳定性

建议按照优先级逐步实施优化，同时保持代码的可维护性和可扩展性。
