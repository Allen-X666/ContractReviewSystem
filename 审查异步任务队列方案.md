# 审查异步任务队列方案 — 解决 LLM 超时问题

## 一、问题分析

### 1.1 报错现象

```
dashscope - ERROR - HTTPSConnectionPool(host='dashscope.aliyuncs.com', port=443): Read timed out. (read timeout=300)
合同审查.app.llm.tongyi_llm - ERROR - _generate 执行失败: HTTPSConnectionPool(host='dashscope.aliyuncs.com', port=443): Read timed out. (read timeout=300)
```

### 1.2 根因分析

| 层级 | 问题 | 影响 |
|------|------|------|
| **LLM 调用层** | dashscope SDK 的 `_generate` 使用同步 HTTP 调用，默认 `read timeout=300s`，无重试机制 | 单次网络抖动直接导致整个审查任务失败 |
| **审查链层** | `RiskAnalysisChain.analyze_risks()` 逐条调用 LLM，每条单独超时则整批失败 | 一条超时导致整批（2条）结果丢失 |
| **任务执行层** | `ReviewWorker._execute_review()` 使用 `asyncio.create_task` 启动后台任务，但超时后直接 `fail_task`，无重试 | 任务不可恢复，用户只能重新发起 |
| **并发控制层** | `Semaphore(1)` 串行处理，单任务阻塞时后续任务排队等待 | 一个超时任务会阻塞所有后续任务 |
| **FastAPI 层** | `/review/start` 同步等待 `fastApiClient.startReview()`，SpringBoot 的 `timeout=120s` 可能不够 | 前端请求可能在 FastAPI 返回前就超时 |

### 1.3 超时传播链路

```
前端 (timeout=60s)
  └→ SpringBoot (timeout=120s)
       └→ FastAPI /review/start
            └→ asyncio.create_task (后台执行)
                 └→ _execute_review
                      └→ _analyze_clauses (逐条分析)
                           └→ RiskAnalysisChain.analyze_risks
                                └→ TongyiLLM._generate
                                     └→ dashscope.Generation.call (timeout=300s) ← 超时发生点
```

## 二、当前架构 vs 目标架构

### 2.1 当前架构

```
┌──────────┐  HTTP   ┌───────────┐  HTTP   ┌──────────────┐
│ 前端 Vue  │ ──────→ │ SpringBoot│ ──────→ │   FastAPI    │
└──────────┘         └───────────┘         └──────┬───────┘
                                                   │
                                          asyncio.create_task
                                                   │
                                                   ▼
                                          ┌─────────────────┐
                                          │  ReviewWorker    │
                                          │  (内存任务队列)   │
                                          │  Semaphore(1)    │
                                          │                  │
                                          │  直接调用 LLM    │← 无重试、无持久化
                                          └─────────────────┘
```

**问题**：
- 任务状态存储在内存中，进程重启后丢失
- 无重试机制，LLM 超时直接失败
- 无并发控制，无法处理多个同时发起的审查
- 无任务优先级，无法区分紧急/普通任务

### 2.2 目标架构

```
┌──────────┐  HTTP   ┌───────────┐  HTTP   ┌──────────────┐
│ 前端 Vue  │ ──────→ │ SpringBoot│ ──────→ │   FastAPI    │
└──────────┘         └───────────┘         └──────┬───────┘
                                                   │
                                          1. 创建任务(状态=pending)
                                          2. 放入任务队列
                                          3. 立即返回 reviewId
                                                   │
                                                   ▼
                              ┌──────────────────────────────────┐
                              │         Redis 任务队列            │
                              │  ┌─────────┬─────────┬────────┐  │
                              │  │ 任务 A  │ 任务 B  │ 任务 C │  │
                              │  │ pending │ pending │ pending│  │
                              │  └─────────┴─────────┴────────┘  │
                              └───────────────┬──────────────────┘
                                              │
                                    ┌─────────┴─────────┐
                                    ▼                   ▼
                           ┌──────────────┐    ┌──────────────┐
                           │  Worker #1   │    │  Worker #2   │
                           │              │    │              │
                           │  取任务      │    │  取任务      │
                           │  执行审查    │    │  执行审查    │
                           │  重试机制    │    │  重试机制    │
                           │  更新进度    │    │  更新进度    │
                           └──────────────┘    └──────────────┘
```

#### 3.2.2 超时配置优化

**文件**: `app/core/config.py` 新增配置项

```python
# LLM 超时与重试配置
LLM_REQUEST_TIMEOUT: int = 120          # 单次 LLM 请求超时（秒），从 300 降到 120
LLM_MAX_RETRIES: int = 3                # 最大重试次数
LLM_RETRY_BASE_DELAY: float = 2.0      # 重试基础延迟（秒）
LLM_RETRY_MAX_DELAY: float = 30.0      # 重试最大延迟（秒）
LLM_BATCH_TIMEOUT: int = 200           # 单批次总超时（秒）
```

**说明**：
- 单次请求超时从 300s 降到 120s，避免长时间等待无效响应
- 采用指数退避重试，避免短时间内大量重试导致限流
- 单批次总超时作为兜底，防止重试累计耗时过长

#### 3.2.3 TongyiLLM 重试改造

**文件**: `app/llm/tongyi_llm.py` 改造 `_generate` 方法

```
改造要点：
1. 在 dashscope 调用前设置超时参数
2. 使用 tenacity 库实现指数退避重试
3. 区分可重试异常（超时、连接错误）和不可重试异常（认证失败、参数错误）
4. 每次重试记录日志
```

**改造逻辑**：

```
_generate 方法改造流程：
│
├─ 1. 尝试调用 dashscope API
│     ├─ 设置 timeout = LLM_REQUEST_TIMEOUT
│     └─ 成功 → 返回结果
│
├─ 2. 捕获异常
│     ├─ TimeoutError / ConnectionError → 可重试
│     │     ├─ 重试次数 < MAX_RETRIES → 等待后重试
│     │     └─ 重试次数 >= MAX_RETRIES → 抛出异常
│     │
│     ├─ 认证错误 / 参数错误 → 不可重试
│     │     └─ 直接抛出异常
│     │
│     └─ 其他异常 → 记录日志，抛出异常
│
└─ 3. 所有重试失败 → 记录最终错误，抛出异常
```

**重试策略**：
```
第 1 次重试：等待 2 秒
第 2 次重试：等待 4 秒
第 3 次重试：等待 8 秒（不超过 MAX_DELAY=30s）
```

#### 3.2.4 条款级别容错

**文件**: `app/services/review_worker.py` 改造 `analyze_batch` 函数

```
改造要点：
1. 单条条款分析失败时，记录错误并跳过，不影响同批次其他条款
2. 单批次失败时，记录错误并跳过，不影响其他批次
3. 最终结果合并所有成功批次的结果
4. 即使所有批次都失败，任务也不标记为"失败"，而是标记为"部分完成"
```

**改造逻辑**：

```
_analyze_clauses 改造流程：
│
├─ 遍历每个批次
│     │
│     ├─ 尝试分析当前批次
│     │     ├─ 成功 → 收集风险项
│     │     └─ 失败 → 记录错误日志，继续下一批次
│     │
│     └─ 更新进度（无论成功失败）
│
├─ 汇总结果
│     ├─ 全部成功 → 标记完成
│     ├─ 部分成功 → 标记完成，附加警告信息
│     └─ 全部失败 → 标记失败
│
└─ 返回所有成功批次的风险项
```

#### 3.2.5 dashscope 超时设置

**关键改造点**：dashscope SDK 的 `Generation.call()` 和 `MultiModalConversation.call()` 支持以下超时参数：

```python
# 当前调用方式（无显式超时控制）
response = dashscope.Generation.call(
    model=self.model,
    messages=messages_list,
    temperature=self.temperature,
    max_tokens=self.max_tokens,
    top_p=self.top_p,
)

# 改造后调用方式
response = dashscope.Generation.call(
    model=self.model,
    messages=messages_list,
    temperature=self.temperature,
    max_tokens=self.max_tokens,
    top_p=self.top_p,
    timeout=LLM_REQUEST_TIMEOUT,  # 显式设置超时
)
```

> **注意**：如果 dashscope SDK 版本不支持 `timeout` 参数，需要通过 `httpx` 或 `requests` 的全局配置来控制：
>
> ```python
> import dashscope
> dashscope.http_client.timeout = LLM_REQUEST_TIMEOUT
> ```

#### 3.3.2 Redis 任务队列设计

**Redis 数据结构**：

```
# 任务详情（Hash）
review:task:{review_id}
  ├── status:      "pending" | "processing" | "completed" | "failed" | "cancelled"
  ├── stage:       "parsing" | "retrieving" | "analyzing" | "generating"
  ├── progress:    0-100
  ├── message:     "正在分析条款..."
  ├── contract_id: 123
  ├── file_name:   "合同.pdf"
  ├── created_at:  "2026-05-10T15:50:22"
  ├── updated_at:  "2026-05-10T15:51:00"
  └── result:      "{...}" (完成后写入)

# 任务队列（List，FIFO）
review:queue:pending
  ├── review_id_1
  ├── review_id_2
  └── review_id_3

# 任务结果（Hash，完成后写入）
review:result:{review_id}
  ├── risk_items:  "[...]"
  └── average_score: 85.5

# 重试计数（String，TTL）
review:retry:{review_id}
  └── "2" (当前重试次数，TTL=1h)
```

#### 3.3.3 任务生命周期

```
┌─────────┐    入队     ┌─────────┐   取出    ┌────────────┐
│ 创建任务 │ ──────────→│ pending  │ ────────→│ processing │
│ (API)   │            │ (队列中) │          │ (执行中)   │
└─────────┘            └─────────┘          └─────┬──────┘
                                                   │
                              ┌────────────────────┼────────────────────┐
                              ▼                    ▼                    ▼
                       ┌──────────┐         ┌──────────┐        ┌──────────┐
                       │completed │         │  failed  │        │cancelled │
                       │ (完成)   │         │ (失败)   │        │ (取消)   │
                       └──────────┘         └─────┬────┘        └──────────┘
                                                  │
                                            可重试? ──→ 重新入队
                                                  │
                                            超过次数? ──→ 标记最终失败
```

#### 3.3.4 Worker 改造

**改造要点**：

```
ReviewWorker 改造为队列消费者：
│
├─ 启动时
│     ├─ 连接 Redis
│     ├─ 初始化 LLM 链（懒加载）
│     └─ 启动消费循环
│
├─ 消费循环（主循环）
│     ├─ 从 review:queue:pending 取出任务（BRPOP 阻塞等待）
│     ├─ 更新任务状态为 processing
│     ├─ 执行审查流程
│     │     ├─ 成功 → 更新状态为 completed，写入结果
│     │     └─ 失败 → 检查重试次数
│     │           ├─ 未超限 → 重新入队，重试计数 +1
│     │           └─ 已超限 → 更新状态为 failed
│     └─ 继续取下一个任务
│
└─ 关闭时
      ├─ 等待当前任务完成
      └─ 关闭 Redis 连接
```

**并发控制**：

```python
# 使用 asyncio.Semaphore 控制并发审查任务数
MAX_CONCURRENT_REVIEWS = 2  # 同时执行的最大审查任务数

# Worker 消费循环伪代码
async def consume_loop():
    semaphore = asyncio.Semaphore(MAX_CONCURRENT_REVIEWS)
    while running:
        review_id = await redis.brpop("review:queue:pending", timeout=5)
        if review_id:
            asyncio.create_task(
                execute_with_semaphore(semaphore, review_id)
            )
```

#### 3.3.5 任务重试策略

```
重试配置：
├─ 最大重试次数：2（共执行 3 次）
├─ 重试延迟：指数退避
│     ├─ 第 1 次重试：等待 10 秒
│     └─ 第 2 次重试：等待 30 秒
├─ 可重试异常：
│     ├─ TimeoutError（LLM 超时）
│     ├─ ConnectionError（网络连接错误）
│     └─ dashscope 网络相关异常
└─ 不可重试异常：
      ├─ 认证失败（API Key 无效）
      ├─ 参数错误（模型名称错误）
      └─ 余额不足
```

#### 3.3.6 API 层改造

**文件**: `app/api/v1/endpoints/review.py` 改造 `start_review` 端点

```
改造前：
  start_review → 创建任务 → asyncio.create_task(执行审查) → 返回
  （审查在当前进程内执行，进程重启则丢失）

改造后：
  start_review → 创建任务(状态=pending) → 入队 Redis → 返回
  （审查由独立 Worker 从队列消费执行，进程重启后 Worker 重新连接即可恢复）
```

### 3.4 第三层：多 Worker 扩展（P2）

#### 3.4.1 多进程 Worker

```
使用 multiprocessing 或独立进程启动多个 Worker：

主进程 (FastAPI)
  ├── Worker 进程 #1 (消费 review:queue:pending)
  ├── Worker 进程 #2 (消费 review:queue:pending)
  └── Worker 进程 #3 (消费 review:queue:pending)

每个 Worker 独立消费队列，互不干扰。
Redis 的 BRPOP 保证同一个任务只被一个 Worker 消费。
```

#### 3.4.2 可选：Celery 替代方案

如果后续需要更复杂的任务调度（定时任务、任务优先级、任务链），可以引入 Celery：

```
优势：
├─ 成熟的任务队列框架
├─ 内置重试、限流、任务链
├─ 支持 Redis/RabbitMQ 作为 Broker
└─ 监控工具（Flower）

劣势：
├─ 增加系统复杂度
├─ 需要额外维护 Celery Worker 进程
└─ 对当前场景可能过重
```

**建议**：当前阶段使用自研 Redis 队列 + Worker 消费者模式即可，后续如果需求复杂化再考虑 Celery。

## 六、效果预期

| 指标 | 改造前 | 改造后 |
|------|--------|--------|
| 单次 LLM 超时 | 直接失败 | 自动重试 3 次（指数退避） |
| 任务持久化 | 内存，进程重启丢失 | Redis，持久化存储 |
| 并发处理 | Semaphore(1) 串行 | 多 Worker 并行消费 |
| 单条款失败 | 整批失败 | 跳过失败条款，继续处理 |
